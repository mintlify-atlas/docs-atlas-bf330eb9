---
title: "Quickstart"
description: "Deploy your first OpenTelemetry Collector and enable automatic instrumentation in minutes."
---

This guide walks you through the minimal steps to get the OpenTelemetry Operator running in your cluster, deploy a Collector, and enable automatic instrumentation for a workload.

<Warning>
  The OpenTelemetry Collector configuration format is still evolving. The examples below reflect the current stable format, but changes may be required in future versions to stay compatible with the latest Collector image. Pin your Collector image version if you need stability.
</Warning>

<Steps>
  <Step title="Install cert-manager">
    The operator requires cert-manager to manage TLS certificates for its admission webhooks. If you do not already have cert-manager installed, apply the latest release manifest:

    ```bash
    kubectl apply -f https://github.com/cert-manager/cert-manager/releases/latest/download/cert-manager.yaml
    ```

    Wait for cert-manager to be fully ready before proceeding:

    ```bash
    kubectl wait --for=condition=Available deployment --all -n cert-manager --timeout=120s
    ```
  </Step>

  <Step title="Install the operator">
    Apply the OpenTelemetry Operator manifest:

    ```bash
    kubectl apply -f https://github.com/open-telemetry/opentelemetry-operator/releases/latest/download/opentelemetry-operator.yaml
    ```

    Verify the operator is running:

    ```bash
    kubectl get deployment -n opentelemetry-operator-system
    ```

    The `READY` column should show `1/1` before you continue.
  </Step>

  <Step title="Deploy a Collector">
    Create an `OpenTelemetryCollector` resource. This example deploys a Collector that receives traces over OTLP (gRPC and HTTP) and exports them to the console via the `debug` exporter:

    ```bash
    kubectl apply -f - <<EOF
    apiVersion: opentelemetry.io/v1beta1
    kind: OpenTelemetryCollector
    metadata:
      name: simplest
    spec:
      config:
        receivers:
          otlp:
            protocols:
              grpc:
                endpoint: 0.0.0.0:4317
              http:
                endpoint: 0.0.0.0:4318
        processors:
          memory_limiter:
            check_interval: 1s
            limit_percentage: 75
            spike_limit_percentage: 15

        exporters:
          debug: {}

        service:
          pipelines:
            traces:
              receivers: [otlp]
              processors: [memory_limiter]
              exporters: [debug]
    EOF
    ```

    This creates an `OpenTelemetryCollector` named `simplest`. The operator creates a `Deployment` and a `Service` for it automatically.

    Verify the Collector is running:

    ```bash
    kubectl get otelcol
    ```

    Expected output:

    ```
    NAME       MODE         VERSION   READY   AGE
    simplest   deployment   0.148.0   1/1     30s
    ```

    <Note>
      The operator does not fully validate the Collector configuration. If the configuration is invalid, the resource will be created but the underlying Collector pod may crash on startup.
    </Note>
  </Step>

  <Step title="Enable auto-instrumentation">
    Create an `Instrumentation` resource that configures the SDK and exports telemetry to your Collector:

    ```bash
    kubectl apply -f - <<EOF
    apiVersion: opentelemetry.io/v1alpha1
    kind: Instrumentation
    metadata:
      name: my-instrumentation
    spec:
      exporter:
        endpoint: http://simplest-collector:4317
      propagators:
        - tracecontext
        - baggage
        - b3
      sampler:
        type: parentbased_traceidratio
        argument: "0.25"
      python:
        env:
          # Python autoinstrumentation uses http/proto by default,
          # so data must be sent to 4318 instead of 4317.
          - name: OTEL_EXPORTER_OTLP_ENDPOINT
            value: http://simplest-collector:4318
      dotnet:
        env:
          # DotNet autoinstrumentation uses http/proto by default.
          - name: OTEL_EXPORTER_OTLP_ENDPOINT
            value: http://simplest-collector:4318
      go:
        env:
          # Go autoinstrumentation uses http/proto by default.
          - name: OTEL_EXPORTER_OTLP_ENDPOINT
            value: http://simplest-collector:4318
    EOF
    ```

    Verify the resource was created:

    ```bash
    kubectl get otelinst
    ```

    Now annotate your pod (or its `PodTemplate` in a Deployment) to inject instrumentation. The annotation to use depends on your application's language:

    **Java:**

    ```yaml
    annotations:
      instrumentation.opentelemetry.io/inject-java: "true"
    ```

    **Node.js:**

    ```yaml
    annotations:
      instrumentation.opentelemetry.io/inject-nodejs: "true"
    ```

    **Python:**

    ```yaml
    annotations:
      instrumentation.opentelemetry.io/inject-python: "true"
    ```

    **.NET:**

    ```yaml
    annotations:
      instrumentation.opentelemetry.io/inject-dotnet: "true"
    ```

    For example, to instrument a Java application via a Deployment:

    ```yaml
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: my-java-app
    spec:
      selector:
        matchLabels:
          app: my-java-app
      template:
        metadata:
          labels:
            app: my-java-app
          annotations:
            instrumentation.opentelemetry.io/inject-java: "true"
        spec:
          containers:
            - name: app
              image: my-java-app:latest
    ```

    The operator's mutating webhook injects an init container that downloads the instrumentation agent and sets the required environment variables before your application container starts. No changes to your application code or image are needed.

    The annotation value can be:

    | Value | Behavior |
    |---|---|
    | `"true"` | Use the `Instrumentation` CR from the same namespace |
    | `"my-instrumentation"` | Use a named `Instrumentation` CR in the same namespace |
    | `"other-ns/my-instrumentation"` | Use a named `Instrumentation` CR in another namespace |
    | `"false"` | Do not inject |
  </Step>
</Steps>
