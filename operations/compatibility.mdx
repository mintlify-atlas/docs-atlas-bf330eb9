## Go version policy

When used as a Go library or custom distribution, the OpenTelemetry Operator follows the
[official Go release policy](https://go.dev/doc/devel/release#policy), which supports the two
most recent major Go versions.

Support is updated as follows:

- The first operator release after a new Go minor version **N** is published adds build and test
  steps for that version.
- The first operator release after a new Go minor version **N** removes support for Go version
  **N-2**.

Removing support for an unsupported Go version is **not** considered a breaking change.
Official operator binaries may be built with any currently supported Go version.

## Kubernetes version support

The operator aims to support the widest practical range of Kubernetes versions:

- All versions currently maintained by the upstream Kubernetes project (see the
  [Kubernetes releases page](https://kubernetes.io/releases/)).
- All versions offered by popular distributions and hosted platforms, including
  [OpenShift](https://access.redhat.com/support/policy/updates/openshift) and
  [AWS EKS](https://docs.aws.amazon.com/eks/latest/userguide/kubernetes-versions.html).

When a Kubernetes version is removed from support, at least one month's notice is given
beforehand.

## Operator and collector versioning

The operator tracks the same minor version as the OpenTelemetry Collector. For example, operator
`v0.18.1` manages collector `0.18.0`. The patch component of the operator version reflects the
operator's own patch level.

When a custom `spec.image` is configured on an `OpenTelemetryCollector` resource, the operator
does not manage versioning for that resource. In this case, keep the operator version aligned with
the collector version embedded in your custom image.

## cert-manager

cert-manager **v1** is required. The operator uses cert-manager for webhook certificate
management. See the
[cert-manager installation docs](https://cert-manager.io/docs/installation/) for setup
instructions.
